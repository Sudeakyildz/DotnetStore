Yönetici özeti

Çözdüğü problem: Ürün grupları, ürünler, özellik değerleri ve güncel fiyatın tek yerden yönetilmesi. Dağınık tablo veya manuel süreç yerine tek API ve panel.

Kim kullanır: İç operasyon veya yönetici rolü; staj veya küçük ölçekli katalog senaryoları için uygundur.

Neden önemli: Veri bütünlüğü (FK, validasyon), tekrar kullanılabilir servis katmanı ve ileride vitrin veya mobil istemci eklenebilecek net bir REST sözleşmesi sunar.

Kalıcılık SQL Server üzerinden; geliştirmede açılışta migrasyon ve isteğe bağlı örnek veri yüklemesi çalışır.

Kapsam

Bu belge mimari özet, veritabanı ilişkileri, REST API yüzeyi, güvenlik modeli ve kurulum adımlarını içerir. Davranış ve sözleşme ayrıntıları için kaynak kod esas alınmalıdır.

Kapsam dışı: iş analizi, pazar gereksinimleri, ayrıntılı arayüz kullanım kılavuzu.

Sistem mimarisi

HTTP istek akışı: İstemci JSON gövdesi veya sorgu parametreleri ile REST uç noktasına istek atar. ASP.NET Core pipeline isteği ilgili controller’a yönlendirir. Korunan uçlarda önce JWT doğrulanır; ardından FluentValidation istek DTO’sunu kontrol eder. Controller ince kalır; iş kuralı ilgili service sınıfında çalışır. Service, EF Core üzerinden veritabanına yazar veya okur.

Kimlik doğrulama akışı: POST /api/Auth/login başarılı olunca API erişim jetonu (JWT) döner. İstemci sonraki isteklerde Authorization: Bearer <token> başlığını gönderir. API, yapılandırılmış imza ve süre ile jetonu doğrular; geçersiz veya süresi dolmuş jeton 401 üretir.

Geliştirmede tarayıcıdaki React uygulaması, Vite’ın /api vekili ile API’ye ulaşır. Üretimde genelde aynı kök veya reverse proxy kullanılır.

┌──────────────────┐ proxy: /api ┌──────────────────────────┐ │ React (Vite) │ ───────────────────► │ ASP.NET Core Web API │ │ Panel │ JSON + JWT Bearer │ Controllers / Services │ └──────────────────┘ └─────────────┬────────────┘ │ EF Core ▼ ┌──────────────────────────┐ │ Microsoft SQL Server │ └──────────────────────────┘

Veri modeli

Model, katalog (kategori, ürün), esnek özellikler (feature + ürün başına değer) ve fiyat geçmişine yer açacak şekilde ayrılmıştır. Ürün kartında tek bir fiyat sütunu tutmak yerine ProductPrices kullanılır: böylece indirim, tarih aralığı ve ileride geçmiş fiyat izi aynı yapıda kalır. Güncel satır, EndDate boş olan kayıt ve buna bağlı filtreli tekil indeks ile kısıtlanır.

ProductFeatures ara tablosu, ürün başına dinamik özellik (renk, depolama vb.) tekrarını önler; özellik tanımı Features tablosunda merkezi kalır.

REST API

Ön ek: /api. Auth uçları kimlik doğrulaması gerektirmez; diğer controller’lar geçerli JWT ister.

AuthController — /api/Auth

Harici istemcinin kullanıcı oluşturması ve oturum açması için. Login yanıtında süre ve token bilgisi döner; panel bu token’ı saklayıp sonraki çağrılarda kullanır.

HTTPYolİşlev
POST/api/Auth/registerYeni StoreUser; parola hash’lenir
POST/api/Auth/loginKimlik doğrulama; JWT üretimi

CategoriesController — /api/Categories

Ürün gruplarının listelenmesi ve CRUD işlemleri. Silme işlemi yumuşak silme (IsDeleted) olarak uygulanır; listeler genelde aktif kayıtları döner.

HTTPYolİşlev
GET/api/CategoriesTüm gruplar
GET/api/Categories/{id}Tekil grup
POST/api/CategoriesOluşturma
PUT/api/Categories/{id}Güncelleme
DELETE/api/Categories/{id}Yumuşak silme

FeaturesController — /api/Features

Ürün kartlarına bağlanacak özellik şablonları (ör. renk, depolama). İsim benzersizdir; ürün formları bu listeyi doldurmak için okur.

HTTPYolİşlev
GET/api/FeaturesTüm özellikler
GET/api/Features/{id}Tekil özellik
POST/api/FeaturesOluşturma
PUT/api/Features/{id}Güncelleme
DELETE/api/Features/{id}Silme

ProductsController — /api/Products

Ürün yaşam döngüsü ve arama. Liste uç noktası metin, kategori ve fiyat aralığı ile süzülür. Detay, özellik değerleri ve güncel fiyatı birlikte döner. Oluşturma ve güncelleme, fiyat ve özellik satırlarını servis kurallarıyla tutarlı şekilde işler.

POST/api/ProductsOluşturma (fiyat ve özellikler dahil)
PUT/api/Products/{id}Güncelleme
DELETE/api/Products/{id}Silme

Kimlik doğrulama ve güvenlik

  • JWT: İmzalama anahtarı, issuer ve audience appsettings.json içinde yapılandırılır. API her istekte jetonun imzasını, süresini ve hedef kitlesini doğrular.
  • Parola: Düz metin saklanmaz; ASP.NET Core PasswordHasher<StoreUser> ile hash üretilir.
  • Ağ: Geliştirmede HTTP kullanılabilir; üretimde HTTPS zorunlu kabul edilmelidir.
  • CORS: Şu anki politika geliştirme kolaylığı içindir. Canlı ortamda yalnızca güvenilen kökenlere izin verilmelidir.
  • Sır yönetimi: Depoda bulunan imzalama anahtarı örnektir; dağıtımda ortam değişkeni veya gizli kasa kullanılmalıdır.
Özet: Kimlik doğrulama stateless JWT ile yapılır; yetkilendirme ilke bazında genişletilebilir (şu an korunan controller’lar [Authorize] ile işaretlenmiştir).

Ön yüz

Durum ve oturum: Global state kütüphanesi yoktur; sayfa bileşenleri React useState / useEffect ile veri çeker. Giriş sonrası JWT, localStorage üzerinde tutulur; apiFetch her istekte Authorization: Bearer ekler.

Yönlendirme: React Router ile korumalı rotalar RequireAuth ile sarılır. Oturum yoksa giriş sayfasına yönlendirilir.

API adresi: Geliştirmede Vite, /api yolunu API sunucusuna vekil eder (hedef: http://127.0.0.1:5198, böylece localhost IPv6 çözümlemesiyle oluşan bağlantı hataları azaltılır).

Kurulum

Ön koşul: .NET SDK (hedef net9.0), Node.js LTS, SQL Server veya LocalDB.

Backend kurulumu

backend/DotnetStore.Api/appsettings.json içinde ConnectionStrings:DefaultConnection değerini ortamınıza göre ayarlayın.

cd backend/DotnetStore.Api
dotnet run --launch-profile http

API taban adresi: http://localhost:5198. Swagger: http://localhost:5198/swagger (Development).

Frontend kurulumu

cd Frontend
npm install
npm run dev

Vite çıktısındaki yerel URL’yi tarayıcıda açın; API vekili aynı makinede çalışan backend’e yönlendirilir.

Örnek veri (seeder)

Development’ta migrasyonlardan sonra DatabaseSeeder çalışır. Eksik admin kullanıcısı, temel kategoriler, özellikler ve isimleri sistemde olmayan örnek ürünler tamamlanır (idempotent).

Yerel deneme hesabı: kullanıcı admin, parola admin123 — üretimde kullanılmamalıdır.

Geliştirme notları

  • Şema değişikliği: EF migrasyonları db/StajDb/Migrations/ altında; dotnet ef komutları startup projesi olarak API ile çalıştırılmalıdır.
  • Eski şema çakışması: Development’ta belirli SQL hatalarında veritabanı silinip migrasyon yeniden uygulanabilir (Program.cs).
  • Hata yanıtları: Servis katmanı sonuçları ToActionResult ile HTTP kodlarına eşlenir.
